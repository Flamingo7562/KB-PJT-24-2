# 프로젝트 관리 가이드

이 문서는 KB-PJT-24-2 레포지토리의 GitHub 기반 협업 규칙을 정의합니다.

## 기본 원칙

- 모든 작업은 이슈에서 시작합니다.
- 하나의 이슈는 하나의 명확한 결과물을 목표로 합니다.
- PR은 반드시 관련 이슈를 연결합니다.
- React, Spring Boot, JPA는 사용하지 않습니다.

## 작업 흐름

| 단계        | 담당        | 기준                                                |
| ----------- | ----------- | --------------------------------------------------- |
| 이슈 생성   | 작업 제안자 | 이슈 템플릿을 사용하고 배경, 범위, 완료 조건을 작성 |
| 이슈 정리   | 팀          | 우선순위, 영역, 담당자, Project 상태 지정           |
| 브랜치 생성 | 담당자      | 이슈 번호가 포함된 브랜치명 사용                    |
| 구현        | 담당자      | 작은 커밋 단위로 진행                               |
| PR 생성     | 담당자      | PR 템플릿 작성, 관련 이슈 연결                      |
| 리뷰        | 리뷰어      | 동작, 영향 범위, 테스트, 컨벤션 확인                |
| 머지        | 팀          | 승인 및 검증 완료 후 머지                           |
| 종료        | 담당자      | Project 상태를 Done으로 이동                        |

## 이슈 작성 규칙

이슈 제목은 다음 형식을 권장합니다.

```text
feat: [FE] 로그인 화면 구현
fix: [BE] 세션 만료 시 리다이렉트 오류
docs: [GITHUB] GitHub Projects 필드 구성
```

이슈 본문에는 다음 항목이 있어야 합니다.

- 배경: 왜 필요한 작업인지
- 범위: 무엇을 하고 무엇을 하지 않는지
- 완료 조건: PR 머지 전에 확인할 수 있는 기준
- 영향 영역: frontend, backend, database, docs, infra 중 하나
- 참고 자료: 화면 설계, API 명세, 관련 이슈

자세한 작성 예시는 [이슈 작성 템플릿 가이드](ISSUE_WRITING_GUIDE.md)를 참고합니다.

## 라벨 규칙

| 분류     | 라벨 예시                                                      | 의미                     |
| -------- | -------------------------------------------------------------- | ------------------------ |
| 종류     | `type: feature`, `type: bug`, `type: task`                     | 작업 성격                |
| 상태     | `status: triage`, `status: blocked`                            | 정리 필요 또는 진행 불가 |
| 영역     | `area: frontend`, `area: backend`, `area: database`            | 영향 범위                |
| 우선순위 | `priority: p0`, `priority: p1`, `priority: p2`, `priority: p3` | 처리 순서                |

## 브랜치 전략

| 브랜치                   | 용도                                         |
| ------------------------ | -------------------------------------------- |
| `main`                   | 항상 배포 가능하거나 제출 가능한 기준 브랜치 |
| `develop`                | 팀이 필요하다고 합의한 경우 통합 개발 브랜치 |
| `feature/12-login-page`  | 기능 개발                                    |
| `fix/23-session-timeout` | 버그 수정                                    |
| `docs/5-project-guide`   | 문서 수정                                    |
| `chore/7-husky-setup`    | 설정, 빌드, 운영 작업                        |

브랜치명은 소문자와 하이픈을 사용하고, 가능하면 이슈 번호를 포함합니다. 브랜치명에는 `#`를 넣지 않고 숫자만 사용합니다.

## PR 규칙

- PR 제목은 커밋 컨벤션과 비슷하게 작성합니다.
- PR 본문에 `Closes #이슈번호`를 적어 이슈가 자동 종료되도록 합니다.
- 변경 범위가 넓다면 기능별로 PR을 나눕니다.
- 리뷰어가 재현할 수 있도록 확인 방법을 적습니다.
- 화면 변경은 스크린샷 또는 짧은 설명을 남깁니다.

## 완료 기준

작업은 다음 조건을 충족해야 Done으로 이동할 수 있습니다.

- 이슈의 완료 조건을 충족했습니다.
- 관련 PR이 승인 및 머지되었습니다.
- 필요한 테스트 또는 수동 검증 결과가 PR에 남아 있습니다.
- Project 패널 상태가 최신입니다.
- 금지 기술인 React, Spring Boot, JPA가 추가되지 않았습니다.
