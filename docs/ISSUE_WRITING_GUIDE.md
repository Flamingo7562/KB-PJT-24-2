# 이슈 작성 템플릿 가이드

이슈 제목, 커밋 메시지, PR 제목은 같은 형식을 사용합니다.

```text
<type>: [AREA] <summary>
```

## 제목 예시

| 작업 | 이슈 제목 |
| --- | --- |
| 프론트엔드 기능 | `feat: [FE] 로그인 화면 구현` |
| 백엔드 기능 | `feat: [BE] 로그인 API 구현` |
| DB 작업 | `feat: [DB] 사용자 조회 mapper 추가` |
| 버그 수정 | `fix: [BE] 세션 만료 시 리다이렉트 오류` |
| 문서 작업 | `docs: [GITHUB] 이슈 템플릿 정리` |
| Hook 설정 | `chore: [HOOK] Husky 설정 추가` |
| 공통 리팩터링 | `refactor: [COMMON] 예외 응답 구조 정리` |
| 테스트 | `test: [BE] 로그인 서비스 테스트 추가` |

## Type 선택 기준

| type | 사용 시점 |
| --- | --- |
| `feat` | 새로운 기능 또는 화면/API 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서, 이슈/PR 템플릿, 가이드 수정 |
| `refactor` | 동작 변화 없는 구조 개선 |
| `test` | 테스트 추가 또는 수정 |
| `chore` | 설정, Hook, 패키지 관리, 단순 작업 |
| `build` | 빌드 설정 변경 |
| `ci` | CI 설정 변경 |
| `perf` | 성능 개선 |

## Area Tag

| area tag | 의미 |
| --- | --- |
| `[FE]` | Vue.js 프론트엔드 |
| `[BE]` | Spring Framework legacy 백엔드 |
| `[DB]` | MyBatis mapper, SQL, 스키마 |
| `[DOCS]` | 문서 |
| `[GITHUB]` | 이슈, PR, Projects 운영 |
| `[HOOK]` | Git Hook, Husky |
| `[INFRA]` | 배포, 환경, 인프라 설정 |
| `[COMMON]` | 공통 모듈, 공통 정책 |

## 기능 이슈 템플릿

```md
제목: feat: [FE] 로그인 화면 구현

## 배경 / 목적
사용자가 인증 후 개인화된 기능을 사용할 수 있어야 한다.

## 요구사항
- [ ] 로그인 화면을 구현한다.
- [ ] 입력값 유효성 검사를 처리한다.
- [ ] 로그인 성공 시 메인 화면으로 이동한다.

## 완료 조건
- [ ] 정상 로그인 흐름을 확인했다.
- [ ] 실패 케이스의 에러 메시지를 확인했다.
- [ ] 관련 PR에서 검증 결과를 남겼다.

## 관련 이슈
Refs #12
```

## 버그 이슈 템플릿

```md
제목: fix: [BE] 세션 만료 시 리다이렉트 오류

## 재현 절차
1. 로그인한다.
2. 세션이 만료될 때까지 기다린다.
3. 보호된 API를 호출한다.

## 기대 결과
로그인 화면으로 이동한다.

## 실제 결과
500 에러가 발생한다.

## 완료 조건
- [ ] 세션 만료 시 로그인 화면으로 이동한다.
- [ ] 관련 로그 또는 테스트 결과를 PR에 남긴다.
```

## 작업 이슈 템플릿

```md
제목: docs: [GITHUB] Projects 운영 가이드 정리

## 작업 범위
- [ ] Project 필드 정의를 정리한다.
- [ ] Status 이동 기준을 정리한다.

## 완료 조건
- [ ] 팀원이 같은 기준으로 Project 패널을 사용할 수 있다.
- [ ] README에서 문서 링크를 확인할 수 있다.
```

## 이슈와 PR 연결

- 이슈 본문에서 단순 참조는 `Refs #12`를 사용합니다.
- `dev` 대상 작업 PR에서는 `Refs #12`를 사용합니다.
- `main` 대상 배포·긴급 수정 PR에서 자동 종료할 때는 `Closes #12`를 사용합니다.
- 커밋 메시지에는 `feat: [FE] 로그인 화면 구현 (#12)`처럼 이슈 번호를 붙입니다.
- 브랜치명에는 `#`를 넣지 않고 `feature/12-login-page`처럼 숫자만 사용합니다.
